== NAS

[cols="30%,70%",options="header"]
|===
| Update | Description 
a|
xref:../flexcache/index.html[NAS performance enhancement]
a|
ONTAP 9.19.1 allows more CPU resources to process listing requests of small NAS directories. This enhancement benefits customers with hundreds to thousands of clients simultaneously listing the same small NAS directory, sized at 2 MB or less (approximately 25,000 files).

This enhancement reduces the need to use workarounds, such as local FlexCache volumes, to support high client counts performing directory listings. For lower client counts, performance will not be negatively impacted.  This performance enhancement is automatic and does not require configuration.

//ONTAPDOC-3936

== Networking

[cols="30%,70%",options="header"]
|===
| Update | Description 
a| 
Manage the NFS versions that the SVM supports
a| 
NFSv4.2 support is managed independently of NFSv4.1 settings, allowing each SVM to advertise or suppress NFSv4.2 capabilities as needed. By default, NFSv4.2 is enabled when a new NFS service is created unless explicitly disabled by the administrator. 

*  xref:../nfs-admin/enable-disable-nfs-task.html[Enable or disable NFSv4.2 for ONTAP SVMs]

* https://docs.netapp.com/us-en/ontap/task_nas_enable_linux_nfs.html[Enable NAS storage for Linux servers using NFS with ONTAP System Manager]

//ONTAPDOC-3656
a|
xref:../concepts/path-failover-concept.html#nas-path-failover[LIF failover enhancement] 
a| 
The ONTAP system-defined failover group policy dynamically expands the failover targets for NAS data LIFs to potentially include any available ports in the broadcast domain on any available node in the cluster. You can apply the system-defined policy to any failover group. Prior to ONTAP 9.19.1, LIFs with this policy could fail over to only one other node in the cluster; if that node was out of quorum, the LIF would not failover, even if there were other nodes available.

//CONTAP-665888 

a|
link:../networking/improve-tcp-wan-performance.html[TCP performance improvements]
a|
ONTAP improves TCP data transfer performance over lossy, high-latency WAN links by integrating two loss-recovery features, PRR and RACK, into the ONTAP TCP stack. PRR is suitable for all networks and is enabled by default. RACK is suitable for congested networks with link speeds of approximately 25 Gigabits/second and is disabled by default.

//ONTAPDOC-3794

== SAN 

[cols="30%,70%",options="header"]
|===
| Update | Description 
a|
xref:../san-config/configure-fc-nvme-hosts-ha-pairs-reference.html[Support for direct-attachment FC configurations]
a|
FC switches are not needed to connect FC and FC-NVMe hosts to ONTAP storage. Hosts can connect directly to FC adapter ports on AFF, ASA, and FAS systems. This direct attachment is especially useful in remote sites and hybrid environments where simplicity is important. 

//ONTAPDOC-3492
a|
xref:../san-admin/multipath-automatic-lif-failover-support.html[Support for active-active multipathing on AFF systems]
a|
AFF systems support active-active multipathing in SAN environments.  In active-active configurations, all paths within the HA pair that owns the LUN or namespace are active-optimized so that in the event of a storage failover, the host does not need to wait for the ALUA or ANA transition of the failover paths to resume I/O. This reduces time to failover allowing SAN customers to run mission-critical workloads on AFF systems with the same non-disruptive path resiliency and load balancing previously available only on ASA systems.

//ONTAPDOC-3574 

a|
xref:../san-admin/multipath-automatic-lif-failover-support.html[Enhanced support for automatic SAN LIF failover]
a|
Automatic LIF failover is supported on AFF and ASA systems for iSCSI protocol LIFs in active-active and active-active-scsi-only multipath configurations and for NVMe/TCP protocol LIFS in active-active multipath configurations.

//ONTAPDOC-3574 

a|
link:https://docs.netapp.com/us-en/ontap-fli/get-started/learn-about-data-migration.html[Foreign LUN import support for iSCSI protocol^]
a|
Foreign LUN Import (FLI) supports the iSCSI protocol for backend connectivity.

== Security  

[cols="30%,70%",options="header"]
|===
| Update | Description 
a|
ARP workload characteristics removed from System Manager UI
a|
System Manager no longer displays Autonomous Ransomware Protection (ARP) workload characteristics, including baseline and surge statistics. This information has been removed as surge activity was frequently misinterpreted as an actual ransomware attack or as something that required manual adjustment.

ARP behavior and detection are unchanged, and only the System Manager presentation of these details is affected. Surge statistics remain available using ONTAP CLI and ONTAP REST API options for advanced analysis

//ONTAPDOC-3636 - No link required; minor UI change 

a|
xref:../networking/configure-tls-offload.html[TLS offload configuration]
a|
You can configure TLS offload to improve TLS post-handshake performance by leveraging resources on supported Ethernet cards. This feature offloads encryption and decryption, reducing CPU overhead and improving performance.

//ONTAPDOC-3497
a|
xref:../nfs-admin/enable-nfs-over-tls-task.html[Enable NFS over TLS to encrypt NFS traffic in transit]
a|
You can enable NFS over TLS with TLS 1.3 to encrypt NFS traffic in transit on NFS-enabled SVMs. This enhancement helps protect NFS data from interception or tampering on untrusted networks. The release also supports optional per-LIF mutual TLS host authentication and export-policy controls to deny non-TLS TCP access for matching clients.
a|
xref:../anti-ransomware/use-cases-restrictions-concept.html#snapmirror[Autonomous Ransomware Protection support for SnapMirror synchronous and SnapMirror active sync]
a|
ARP/AI protects primary volumes in SnapMirror synchronous (NAS and SAN) relationships on FAS, AFF, and https://docs.netapp.com/us-en/ontap-afx/release-notes/whats-new-9191.html[AFX] systems, and in SnapMirror active sync SAN relationships on AFF and https://docs.netapp.com/us-en/asa-r2/release-notes/whats-new-9191.html[ASA r2] systems. Detection runs on the active read-write primary, where ARP snapshots are created (snapshots are not replicated).

Clusters upgraded to ONTAP 9.19.1 that previously had default ARP enablement suppressed due to synchronous replication relationships will have it automatically restored. A pre-update warning notifies administrators and provides the option to disable it.

// 2026-June-26, ONTAPDOC-3817
// 2026-June-1, ONTAPDOC-3918
// 6-11-26: contap-719516
a|
xref:../authentication-access-control/webauthn-mfa-overview.html[WebAuthn Relying Party domain enhancement]
a|
ONTAP 9.19.1 introduces a new parameter `-rp-domains` in the WebAuthn global settings command, which allows administrators to explicitly restrict the set of permitted Relying Party (RP) domains accepted during WebAuthn (FIDO2) multi-factor authentication in System Manager.
