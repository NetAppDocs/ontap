= Restore data from ONTAP ARP snapshots after a ransomware attack
:icons: font
:imagesdir: ../media/

[.lead]
Autonomous Ransomware Protection (ARP) creates snapshots to protect against a potential ransomware threat. You can use one of these ARP snapshots or another snapshot of your volume to restore data.

.About this task
The ARP creates snapshots with one of the following prepended names:

* `Anti_ransomware_periodic_backup`: Used in ONTAP 9.17.1 and later for snapshots created at regular intervals. For example, `Anti_ransomware_periodic_backup.2025-06-01_1248`.
* `Anti_ransomware_attack_backup`: Used in ONTAP 9.17.1 and later for snapshots created in response to abnormalities. For example, `Anti_ransomware_attack_backup.2025-08-25_1248`.
* `Anti_ransomware_backup`: Used in ONTAP 9.16.1 and earlier with snapshots that are created in response to abnormalities. For example, `Anti_ransomware_backup.2022-12-20_1248`.

To restore from a snapshot other than the `Anti_ransomware` snapshot after a system attack is identified, you must first release the lock on the ARP snapshot.

If no system attack is reported, you must first restore from the `Anti_ransomware` snapshot then complete a subsequent restoration of the volume from the snapshot you choose.

.Before you begin
If you are responding to an abnormal activity warning, link:respond-abnormal-task.html[you must first categorize it as a potential ransomware attack] before proceeding with data restoration.

.Steps

Follow these steps when you need to restore data after ARP detects an abnormality. If you need to classify abnormal activity and clear ARP notices, use the steps in link:respond-abnormal-task.html[Respond to abnormal activity detected by ONTAP ARP].

== Recovery constraints and considerations

* For ONTAP 9.15.1 and earlier, releasing the ARP snapshot lock deletes ARP snapshots immediately. Release the lock only if you do not plan to restore from an ARP snapshot.
* In CLI workflows, releasing the anti-ransomware lock before restoring from earlier snapshots is only required when you use `volume snapshot restore`. It is not required for FlexClone, single-file SnapRestore, or similar methods.
* If the volume is part of a SnapMirror synchronous or SnapMirror active sync SAN relationship in ONTAP 9.19.1 RC and you plan a volume-level restore, quiesce or link:../data-protection/make-destination-volume-writeable-task.html[break the relationship before restore] and then reestablish protection after restore.
* If you restore an ARP-protected volume from a snapshot while it participates in a SnapMirror relationship, manually update all mirror copies after the restore. Otherwise, mirror copies can become unusable and might need to be deleted and recreated.
* For full SnapMirror and ARP interoperability behavior, including snapshot and failover considerations, see link:use-cases-restrictions-concept.html#snapmirror[SnapMirror and ARP interoperability].

[[restore-after-system-attack]]
== Restore after a system attack

Choose one of these flows depending on the snapshot source and situation:

* <<restore-from-arp-snapshot,Restore from an ARP snapshot>>
* <<restore-from-non-arp-snapshot,Restore from a non-ARP snapshot>>

[[restore-from-arp-snapshot]]
=== Restore from an ARP snapshot

[role="tabbed-block"]
====
.System Manager
--
. Select *Storage > Volumes*, then select the volume and *Snapshot Copies*.

. Select image:icon_kabob.gif[Menu options icon] next to the ARP snapshot you want to restore, then select *Restore*.

. After the restore completes, link:respond-abnormal-task.html#arp-snapshot-decision-sm[record your decision and clear notices].
--

.CLI
--
. List the snapshots in a volume:
+
[source,cli]
----
volume snapshot show -vserver <SVM> -volume <volume>
----

. Restore the contents of a volume from a snapshot:
+
[source,cli]
----
volume snapshot restore -vserver <SVM> -volume <volume> -snapshot <snapshot>
----

. After the restore completes, link:respond-abnormal-task.html#arp-snapshot-decision-cli[record your decision and clear notices].
--
====

[[restore-from-non-arp-snapshot]]
=== Restore from a non-ARP snapshot

[role="tabbed-block"]
====
.System Manager
--
. Release the lock on the ARP snapshot:
+
.. Select *Storage > Volumes*.
.. Select *Security* then *View Suspected File Types*.
.. Mark the files as "Potential ransomware attack".
.. Select *Update* and *Clear Suspected File Types*.
+
NOTE: If you already classified the activity as a potential ransomware attack using the steps in link:respond-abnormal-task.html[Respond to abnormal activity detected by ONTAP ARP], you only need to clear the suspected file types here.

. Select *Storage > Volumes*, then select the volume and *Snapshot Copies*.

. Select image:icon_kabob.gif[Menu options icon] next to the non-ARP snapshot you want to restore, then select *Restore*.
--

.CLI
--
. If you are restoring from a non-ARP snapshot using `volume snapshot restore`, mark the attack as potential ransomware (`-false-positive false`) and clear suspect files to release the lock:
+
[source,cli]
----
anti-ransomware volume attack clear-suspect -vserver <svm_name> -volume <vol_name> [<extension identifiers>] -false-positive false
----
+
Use one of the following parameters to identify the extensions:
+
* `[-seq-no _integer_]`: Sequence number of the file in the suspect list.
* `[-extension _text_, … ]`: File extensions
* `[-start-time _date_time_ -end-time _date_time_]`: Starting and ending times for the range of files to be cleared, in the form "MM/DD/YYYY HH:MM:SS".

include::../_include/restore-snapshots-cli.adoc[]
--
====

[[restore-without-attack]]
== Restore when a system attack is not identified

When no attack is identified, first restore from the ARP snapshot and then restore from the snapshot you choose.

[role="tabbed-block"]
====
.System Manager
--
. Select *Storage > Volumes*, then select the volume and *Snapshot Copies*.
. Select image:icon_kabob.gif[Menu options icon] then choose the `Anti_ransomware` snapshot.
. Select *Restore*.
. Return to the *Snapshot Copies* menu, then choose the snapshot you want to use.
. Select *Restore*.
--

.CLI
--
. Restore from the ARP snapshot first:

.. List the snapshots in a volume:
+
[source,cli]
----
volume snapshot show -vserver <SVM> -volume <volume>
----

.. Restore the contents of a volume from a snapshot:
+
[source,cli]
----
volume snapshot restore -vserver <SVM> -volume <volume> -snapshot <snapshot>
----

. Repeat the restore and select the non-ARP snapshot you want to use.

Learn more about `volume snapshot` in the link:https://docs.netapp.com/us-en/ontap-cli/search.html?q=volume+snapshot[ONTAP command reference^].
--
